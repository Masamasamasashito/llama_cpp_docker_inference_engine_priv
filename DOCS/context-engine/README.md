# 機密レベル分離型 Context Engine 設計（LDIE応用）

本ドキュメントは、LDIEの設計思想である「脅威の大半はUbuntu（OpenClaw）側に集中し、Windows（LLMサーバー）側は比較的安全である」という前提を活かし、個人の極めてプライベートな情報を保護するための実運用アーキテクチャを定義する。

目標は次の2点です。

- 外部通信を伴うAIエージェント（OpenClaw）を継続運用できること
- 超機密の文脈エンジンのデータをネットワーク侵害から物理的・論理的に守り抜くこと

---

## 1. 設計思想：文脈の完全分離（Zero Trust）

文脈エンジンのデータ（Obsidian Vault等）を単一保管せず、機密レベルごとに物理分離する。

- **セキュア層（Windows 11 Pro / API提供側）**
  - 超機密情報を含む「フルセット文脈エンジン」を保持
  - 原則としてインターネット非依存
  - 管理者が物理筐体に直接サインインして利用
  - LDIEの組み込みWebUI経由で壁打ち・戦略策定を実施

- **DMZ・エージェント層（Ubuntu 24.04.4 LTS / API利用側）**
  - 外部接続（Discord/インターネット）を伴う OpenClaw を配置
  - 「無害化（パブリック）文脈」のみ配置
  - 技術ノウハウ、公開前提トーン&マナー、公開可能な運用情報のみを保持
  - 超機密情報は保存しない

**重要原則**

- Windows側からUbuntu側へのファイル共有（SMB/NFS等）は行わない
- Ubuntu側からWindows側へのファイル参照経路を作らない
- 許可するのは LLM API 呼び出しのみ

---

## 2. 参照アーキテクチャ

### 2.1 構成要素

- **Windows 11 Pro（VLAN2）**
  - `llama.cpp` サーバー（LDIE）
  - 超機密文脈ファイル保管領域
  - 組み込みWebUI（ローカル利用）

- **Ubuntu 24.04.4 LTS（VLAN1）**
  - OpenClaw Gateway
  - Discord Bot連携
  - パブリック文脈ファイル

- **ルーター（例：YAMAHA RTX1300）**
  - VLAN分割
  - インターネット向け制御
  - VLAN間フィルタ（最小許可）

### 2.2 VLAN分割（例）

- **VLAN1**: Ubuntu（OpenClaw + Discord Bot）
- **VLAN2**: Windows（LDIE + フルセット文脈エンジン）

---

## 3. ネットワーク制御方針（多層防御）

### 3.1 VLAN間許可ルール

VLAN1（Ubuntu）→ VLAN2（Windows）に対して許可する通信を厳格に限定する。

- 許可: `TCP 8081`（または運用ポート）への LLM API 通信のみ
- 拒否: 上記以外すべて

### 3.2 明示的に禁止する代表例

- `TCP 445`（SMB）
- `TCP 139`（NetBIOS Session）
- `TCP/UDP 137-138`（NetBIOS Name/Datagram）
- `TCP 22`（SSH, 不要なら禁止）
- `TCP 3389`（RDP, 不要なら禁止）

### 3.3 二重防御

- ルーター（RTX1300想定）のパケットフィルタで VLAN 間通信を制御
- Windows Defender 受信規則でも同じポリシーを適用（二重化）

### 3.4 本ドキュメントの採用方針（B: 機密保護優先）

本設計では **B. 機密保護優先** を正式方針とする。

- Ubuntu（VLAN1）→ Windows（VLAN2）は `TCP 8081`（LLM API）のみ許可
- `TCP 445/139/137/138`（SMB/NetBIOS）は常時閉鎖
- Windows側の超機密Vaultはネットワーク共有しない
- Ubuntu側からWindows側Vaultのマウント（CIFS/SMB/NFS）は行わない

---

## 4. ファイル配置ポリシー（機密レベル別）

### 4.1 Windows側（超機密）

保存対象（例）:

- 詳細な個人履歴
- 資産/収益情報
- 非公開の事業戦略
- 機微な顧客情報・契約情報

制約:

- Ubuntu側へコピーしない
- 共有フォルダ公開しない
- 外部同期対象から除外

### 4.2 Ubuntu側（無害化）

保存対象（例）:

- 公開前提の運用ノウハウ
- ブログ/発信の文体ガイド
- 一般化済みの技術手順

制約:

- 機密情報を持ち込まない
- 定期的にデータ内容を監査する
- Windows側の超機密Vaultをネットワークマウントしない

### 4.3 連携方式（B方針）

- Ubuntu側OpenClawは、Ubuntuローカルの無害化文脈のみを参照する
- 推論はWindows側LDIE API（`TCP 8081`）を呼び出して実行する
- Windows側のフルセット文脈エンジンは、管理者がWindows物理端末でのみ利用する

---

## 5. 攻撃シナリオと防御効果

### シナリオA: Ubuntu側が侵害された

- 攻撃者が取得できるのは Ubuntu 上のパブリック文脈のみ
- Windows側の超機密文脈にはネットワーク上到達不可（APIポート以外遮断）

### シナリオB: API経由で悪性プロンプト注入

- `llama.cpp` は推論エンジンであり、任意にWindows内ファイルを探索する機構を持たない
- ファイル参照責務はクライアント側（Ubuntu）にある
- Ubuntuに超機密ファイルが存在しないため窃取不能

---

## 6. 運用フロー

### 6.1 日常運用（自走タスク）

1. Ubuntu上のOpenClawが外部チャネル（Discord等）を処理
2. パブリック文脈を参照してタスク実行
3. 推論はWindows側LDIE APIを呼び出して実行

### 6.2 高機密運用（意思決定/戦略）

1. 管理者がWindows物理筐体に直接ログイン
2. LDIE組み込みWebUIをローカル利用
3. フルセット文脈エンジンで深い思考・戦略策定を実施

---

## 7. 実装チェックリスト（初期導入）

- [ ] VLAN1（Ubuntu）と VLAN2（Windows）を分離した
- [ ] RTX1300等で VLAN1→VLAN2 は APIポートのみ許可した
- [ ] Windows Defender 受信規則でも同等制御を設定した
- [ ] SMB/NetBIOS等の共有系ポートを遮断した
- [ ] Ubuntuに超機密ファイルが存在しないことを確認した
- [ ] Windowsにフルセット文脈を保管し共有しない設定にした
- [ ] OpenClaw側は無害化文脈のみを読み込む設定にした

---

## 8. 運用上の注意

- この設計は「被害を局所化する設計」であり、Ubuntu侵害の可能性自体は前提として扱う
- 「データを置かないこと」が最大の防御。Ubuntuへの誤配置が最大リスク
- ポリシー逸脱（例: 一時コピー）を防ぐため、運用手順を文書化して定期監査する
